# 长篇系列小说交互式阅读增强系统需求定义 v0.3

> 2026-06-30 状态覆盖：阶段 5-8 清洗后操作逻辑已由 `docs/post-cleaning-operation-design-v0.2.md` 定案。凡本文仍出现「逐候选人工复核」「AI 不得写 Accepted」「待重构 / 暂停实现」等旧口径，均以 v0.2 为准：AI 起草 + 独立 AI 复核后可自动写 Accepted，但必须生成可追溯、可回滚 Change，高风险项升级给人裁决。

## 1. 项目定位

本项目是面向长篇系列小说的本地交互式阅读增强系统。

基础阅读器只是入口，核心价值是围绕长篇叙事文本提供：

- 防剧透上下文
- 人物卡和术语卡
- 说话人标注
- 事件与关系记忆
- 剧情线回顾
- 数值变化追踪
- AI 辅助但人工复核的数据整理

系统当前以复杂校园群像、特殊考试和长期人物关系变化作为压测方向，但设计目标是通用增强阅读器。作品特有字段应通过数据配置和可选模块表达，不写死为单作品工具。

## 2. 产品边界

长期定位是本地桌面 App。第一阶段可以先用文件夹工程、脚本、最小 Web/桌面阅读器和数据工作台验证闭环。

长期不优先考虑：

- 账号系统
- 云同步
- 在线内容平台
- 自动下载数据包
- 移动端优先
- 浏览器插件优先
- EPUB 原版渲染优先

当前阅读端优先使用清洗后的 Markdown 私有正文格式。EPUB 保留为原始 Source 和未来二期方向。

## 3. 内容与权利边界

软件发行方只发行：

- 软件本体
- 数据格式规范
- 本地解析、校验、编译工具
- AI 提示词和工作流
- 公版、授权或自制样例数据

软件发行方不发行：

- 版权小说原文
- 可还原版权原文的大规模 parsed blocks
- 官方版权作品完整增强数据包
- 集中托管的系列百科数据
- 数据包索引站、订阅源或在线市场

完整阅读包是否可以公开分发，取决于文本和资源权利状态。manifest 必须声明正文、图片、日文参考和使用范围。

## 4. 第一阶段目标

第一阶段目标不是完整桌面应用，也不是发布打包优化，而是跑通完整制作闭环：

1. 定义清洗后数据接受形式。
2. 用原创测试文段验证清洗、解析、候选、复核、入库、编译和阅读展示。
3. 实现最低限度 Markdown 阅读器。
4. 实现最小数据工作台。
5. 验证防剧透查询、block 阅读标尺、右侧增强面板和变化记录。

第一阶段按流程分段测试：

1. 清洗输出模板。
2. Markdown block / scene / asset / alignment 标记。
3. Parsed JSONL 生成。
4. 硬校验。
5. AI Candidates JSONL 模板。
6. AI 起草 + 独立 AI 复核工作台。
7. Accepted / Change / OpenQuestion 入库。
8. Compiled 查询。
9. 最低限度 Markdown 阅读器。

### 4.1 2026-06-30 执行状态更新

阶段 1-4（测试书、清洗 Markdown、Parsed 生成、硬校验）已按当前工具链推进。

阶段 5-8（AI Candidates、复核、Accepted 写入、数据工作台相关操作流）曾在一次候选卡片式 Web 工作台原型验证后暂停。验证结论是：如果按“AI 生成大量候选 -> 人工逐条点击接受/拒绝/转换”的 candidate-centric 模式处理真实长篇小说，制作工作量会过大，交互负担会集中在低层候选微操作上。

当前结论已落到 `docs/post-cleaning-operation-design-v0.2.md`：保留 Candidates 作为起草到复核之间的内部中间格式；由起草 Agent 生成草案、独立复核 Agent 路由，低风险自动写 Accepted + Change，高风险进入异常队列，人负责异常裁决和 Change 审计。

## 5. 数据包类型

系统区分工程包和阅读包。

工程包用于数据制作、AI 清洗、人工复核和维护，可包含：

- Source
- Parsed
- Candidates
- Review
- OpenQuestions
- Reports
- Accepted
- Compiled

阅读包用于普通阅读，只包含稳定阅读数据：

- manifest
- Parsed / 正文资源
- Accepted
- Compiled
- 必要 assets

阅读包默认不包含 Candidates、Review、OpenQuestions 或 AI 中间产物。

第一阶段优先支持包含正文和必要资源的完整阅读包。注释型阅读包和匹配用户本地 EPUB 的能力保留为长期方向。

## 6. 中文主轴与日文参考

系统以中文文本为唯一主轴。

- block、章节、当前位置、防剧透边界、实体来源、事件来源、关系来源均以中文文本为准。
- 日文不建立独立 block 体系。
- 日文只作为中文 block 的参考渲染内容。
- 日文不进入 AI 操作主链路，除非任务明确是中日对齐复核。
- 阅读器可提供中文 / 中文 + 日文参考切换。

## 7. 防剧透原则

防剧透不是 UI 装饰，而是数据查询层硬约束。

系统区分：

- `current_block`：用户当前看到哪里，用于当前位置相关展示。
- `read_boundary`：系统确认读者已安全读完哪里，用于防剧透查询。

所有会影响剧透边界的数据都必须带 `visible_from` 或等价字段。查询时以 `read_boundary` 为准，不直接使用 `current_block` 放宽边界。

## 8. AI 边界

AI 的结构化输出默认先进入 Candidates；不确定、冲突、长期悬而未决或需要后文回查的内容可以进入 ReviewItems 或 OpenQuestions。

AI 输出必须带：

- 来源位置
- 证据范围
- 置信度
- 可见边界
- 任务类型
- 模型信息

v0.2 后，AI 可在**独立复核通过**后自动写入 Accepted，但不能静默写入：每次写入都必须生成 Change，标记 `decided_by`、`auto_accepted`、`reviewer_model`、`work_run_id`，并支持单 Change / 整批 work_run 回滚。实体合并、歧义说话人、关系变化、伏笔/隐藏身份、数值冲突、图片人物身份等高风险项必须升级给人裁决。

## 9. Agent 驱动的数据制作原则

数据工作台不是 AI 聊天入口，而是 Agent 驱动的制作系统。

当前注意：本节描述的是 Agent 的职责边界和长期原则。真实工作台已按 `docs/post-cleaning-operation-design-v0.2.md` 改为作业控制台、异常队列、Change 审计和回滚，而不是按候选卡片逐条复核。

项目不应依赖“每次打开一个 chatbox，手工粘贴大量上下文和 schema，再把结果搬回项目”的工作方式。清洗后的文本操作阶段应由内置制作 Agent 负责协调上下文、工具接口和数据状态。

内置 Agent 应能：

- 读取当前 bookpack 的 manifest、Parsed、Candidates、Review、Accepted 和 Reports。
- 检索当前 block、scene、已确认实体、事实、事件、关系、数值、角色卡和未决问题。
- 调用 parser、validator、candidate generator、review queue、accepted store 和 compiler。
- 为所选章节/范围生成候选、风险提示和建议操作。
- 用独立复核模型核对草案，低风险自动落盘，高风险升级给人工复核。
- 执行结构化写入时同步生成 Change，并提供审计和回滚。

聊天可以作为解释、辅助和调试界面，但主流程必须是结构化任务、候选、复核、写入和审计。人类制作者的核心职责是复核和裁决，而不是反复搬运上下文或手工拼接 JSON。

## 10. 质量原则

- 防剧透优先于信息完整。
- 不确定时宁可不显示，也不乱标。
- 正式增强数据必须可追溯；默认追溯到中文正文 block，图片主体等例外必须能通过 asset anchor 回到正文位置。
- 人工确认结果优先。
- 阅读界面克制，数据工作台可以复杂。
- 数值数据不能靠推测。
- 数据包应能重复生成、重复校验、重复编译。
