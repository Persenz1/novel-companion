# 待接手需求：清洗起点 + 增强交互

本文记录用户 2026-07-01 口述的三条方向性需求，供后续接手者实现。均为“记录待做”，当前未实现。优先级见每条标题。

## 1.（高优先，先做）清洗流程的真正起点：整本 EPUB → 多模态清洗

**现状问题**：当前测试的默认起点是「已经清洗好、已切好 block 的 Markdown + 图片」（即 `samples/gray-tower` 那种 bookpack）。这不该是起点——否则用户得先在系统外把书洗好、切好 block，体验不完整。

**目标流程**：用户直接投喂**一整本 EPUB**，由 MiMo（或其他多模态大模型）在系统内完成清洗（结构切分、block 划分、图片锚点与图注等）。

**已知障碍 / 第一步要做的**：MiMo 无法直接吃 EPUB 文件，所以需要**先写一个 EPUB 解包 / 解析器**：
- 解开 EPUB（本质是 zip：`META-INF/container.xml` → `.opf` spine → 各 XHTML 章节 + `images/`）。
- 抽出正文文本、章节结构、图片二进制及其在正文中的锚点位置。
- 产出可喂给多模态模型的中间物（文本分片 + 图片），再由模型驱动切 block、定图注，落成本仓库的 bookpack 结构（manifest + parsed/volumes + assets）。

**与旧设想的冲突（以本需求为准）**：这与最早「去外面找 Codex / GPT 做清洗，系统只接已洗好的数据」相冲突。**用户明确以现在这条为准**：清洗进系统、多模态模型负责，从原始 EPUB 起步。参见 [[workpack-persistent-and-config]] 里已落的清洗·图片子流程（那是清洗的一部分，但起点仍是已切好的包，需要往前接到 EPUB 解析）。

## 2.（待讨论）已确认 block 右侧看不到增强信息

**现象**：即便是已经做过确认 / 裁决的 block，阅读器 / 工作台右栏依旧看不到增强信息。

**用户态度**：**先不要当 bug 去修**。后续还要先讨论「右栏究竟应该显示什么内容」，再决定怎么改。所以这里只登记现象，不预设是渲染 bug 还是数据 / 边界（`read_boundary` / `visible_from` / Accepted-only）问题。接手前先跟用户对齐「显示什么」。

相关既有约束见 `requirements.md` §4（只展示 Accepted、按 `read_boundary` 过滤）与 [[workbench-image-review-gap]]（图片复核项要在界面显示图）。

## 3.（不急，系统跑通后再做）AI 提取型增强交互 + 预留显示接口

**需求**：人物信息卡、角色事件时间线等——由 AI 从**已经洗好的数据**里提取信息，做成增强交互展示。这类「提取型增强」目前还没做。

**优先级**：不急，可以等整套系统（EPUB 清洗 → 起草 → 复核 → 阅读）跑顺后再做。用户会想到什么加什么。

**当前唯一硬要求**：**留好显示接口**。即现在设计阅读器 / 查询产物 / getVisibleContext 输出时，要给「人物卡 / 事件时间线 / 关系图」等预留展示位与数据挂点，避免以后返工。防剧透仍按 `read_boundary` 走。

---

**一句话交接**：先把清洗起点从「已切好的包」往前推到「整本 EPUB + 多模态清洗」（第 1 条，先写 EPUB 解析器）；第 2、3 条要先跟用户对齐「显示什么」再动手，但接口现在就预留。
