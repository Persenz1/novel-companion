# 待接手需求：清洗起点 + 增强交互

本文记录用户 2026-07-01 口述的三条方向性需求。实现事实以 `cleaning-pipeline.md`、`status.md` 和当前代码为准；本文只保留后续推进方向。

## 1.（高优先，先做）清洗流程的真正起点：整本 EPUB → 多模态清洗

**原始问题**：早期测试的默认起点是「已经清洗好、已切好 block 的 Markdown + 图片」（即 `samples/gray-tower` 那种 bookpack）。这不该是最终起点——否则用户得先在系统外把书洗好、切好 block，体验不完整。

**当前进展**：已经有 EPUB importer、MiMo 任务包、MiMo headless 调用和 `/cleaning/` 一键清洗入口。用户侧可以填写一个或多个 EPUB 路径开始清洗；多个单卷 EPUB 会汇入同一 bookpack。MiMo 建议主要写报告和界面展示，尚未做通用写回。

**目标流程**：用户直接投喂**一整本 EPUB**，由 MiMo（或其他多模态大模型）在系统内完成清洗（结构切分、block 划分、图片锚点与图注等）。

**剩余障碍**：
- 真实多卷 EPUB 拆分：从目录 / 标题 / spine 识别 `v01`、`v02` 等卷；受控多单卷 EPUB append 路径已可用。
- 真实 EPUB 兼容性：脚注、跨文件章节、异常目录、广告页、复杂排版。
- 清洗建议写回：把低风险建议安全应用到 manifest / Markdown / assets，高风险建议进入人工确认。
- 和后处理统一：清洗后起草 / 复核也应从“整本书输入，内部按卷/章处理”的口径推进。

**与旧设想的冲突（以本需求为准）**：这与最早「去外面找 Codex / GPT 做清洗，系统只接已洗好的数据」相冲突。**用户明确以现在这条为准**：清洗进系统、多模态模型负责，从原始 EPUB 起步。

## 2.（待讨论）已确认 block 右侧看不到增强信息

**现象**：即便是已经做过确认 / 裁决的 block，阅读器 / 工作台右栏依旧看不到增强信息。

**用户态度**：**先不要当 bug 去修**。后续还要先讨论「右栏究竟应该显示什么内容」，再决定怎么改。所以这里只登记现象，不预设是渲染 bug 还是数据 / 边界（`read_boundary` / `visible_from` / Accepted-only）问题。接手前先跟用户对齐「显示什么」。

相关既有约束见 `requirements.md` §4（只展示 Accepted、按 `read_boundary` 过滤）。

## 3.（不急，系统跑通后再做）AI 提取型增强交互 + 预留显示接口

**需求**：人物信息卡、角色事件时间线等——由 AI 从**已经洗好的数据**里提取信息，做成增强交互展示。这类「提取型增强」目前还没做。

**优先级**：不急，可以等整套系统（EPUB 清洗 → 起草 → 复核 → 阅读）跑顺后再做。用户会想到什么加什么。

**当前唯一硬要求**：**留好显示接口**。即现在设计阅读器 / 查询产物 / getVisibleContext 输出时，要给「人物卡 / 事件时间线 / 关系图」等预留展示位与数据挂点，避免以后返工。防剧透仍按 `read_boundary` 走。

---

**一句话交接**：清洗起点已经推进到「EPUB + 多模态清洗」MVP；下一步是多卷、真实 EPUB、建议写回和右栏展示口径。第 2、3 条要先跟用户对齐「显示什么」再动手，但接口现在就预留。
