# 阶段 5-8 清洗后数据操作逻辑重构说明

状态：待讨论，暂停实现。

## 背景

本轮曾尝试实现清洗后数据操作阶段的 Agent、候选生成和 Web 工作台，用于覆盖：

- AI Candidates 生成
- block 复核
- Accepted 写入
- ReviewItem / OpenQuestion 转换
- Compiled 前的数据操作验证

实际交互验证后发现，如果按“AI 生成大量候选 -> 人工逐条点选接受/拒绝/转换”的模式推进，真实长篇小说的数据制作工作量会非常夸张，且操作负担会集中到候选级微操作上。

因此，本轮实现已回滚。阶段 5-8 的操作逻辑需要重新讨论后再继续。

## 暂停范围

以下模块暂不按当前方案继续实现：

- 内置制作 Agent 的操作循环
- CandidateGenerator 的实际交互形态
- Web 数据工作台
- block 复核队列的 UI/UX
- AI 候选到人工确认的批处理方式
- Accepted 写入前的人机协作流程

## 保留原则

重新设计时仍应遵守既有硬边界：

- 中文正文是唯一主轴。
- AI 不能绕过人工确认写 Accepted。
- Accepted 必须可追溯，并生成 Change。
- 防剧透查询仍以 `read_boundary` 为准。
- 工作台复核最小单位仍优先考虑 block，但不等于每个候选都必须逐条手工点击。
- 第一阶段继续优先使用 JSON/JSONL 和 Markdown。

## 待讨论问题

- 人工复核的真实最小操作单位是否应从“candidate”上移到“block 批处理决策”或“scene / range 批处理建议”。
- AI 输出是否应先聚合成 block-level digest，再由人工一次性确认一组低风险结构化写入。
- 哪些数据类型可以批量确认，哪些必须单条确认。
- 如何设计“自动草稿 -> 人工批注 -> Agent 结构化落盘”的低摩擦流程。
- 工作台是否应以任务流、差异视图、批处理确认和异常队列为核心，而不是候选卡片列表。
- 如何展示上下文预算和作业范围，使其服务决策而不是变成调试噪声。
