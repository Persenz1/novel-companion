# 阶段 5-8 清洗后数据操作逻辑重构说明

状态：已定案。本文记录回滚背景与待讨论问题；结论已落到 `docs/post-cleaning-operation-design-v0.2.md`（AI 驱动 + 独立 AI 复核 + 人审计异常）。下文的「待讨论问题」保留为历史脉络，答案以 v0.2 为准。

## 背景

本轮曾尝试实现清洗后数据操作阶段的 Agent、候选生成和 Web 工作台，用于覆盖：

- AI Candidates 生成
- block 复核
- Accepted 写入
- ReviewItem / OpenQuestion 转换
- Compiled 前的数据操作验证

实际交互验证后发现，如果按“AI 生成大量候选 -> 人工逐条点选接受/拒绝/转换”的模式推进，真实长篇小说的数据制作工作量会非常夸张，且操作负担会集中到候选级微操作上。

因此，本轮实现已回滚。阶段 5-8 的操作逻辑需要重新讨论后再继续。

## 暂停范围（历史；已由 v0.2 解除并重新定义）

旧方案下以下模块曾暂停实现，现已按 `docs/post-cleaning-operation-design-v0.2.md` 重新设计：

- 内置制作 Agent 的操作循环
- CandidateGenerator 的实际交互形态
- Web 数据工作台
- block 复核队列的 UI/UX
- AI 候选到人工确认的批处理方式
- Accepted 写入前的人机协作流程

## 保留原则

既有硬边界（注：部分已在 v0.2 中有意识修订，下方标注）：

- 中文正文是唯一主轴。
- ~~AI 不能绕过人工确认写 Accepted。~~ → v0.2 修订：AI 经独立复核后可自动写 Accepted，但每次写入是可回滚 Change，高风险类别升级给人。
- Accepted 必须可追溯，并生成 Change。（保留，且更重要）
- 阅读器查询仍以 `read_boundary` 为准；但防剧透不是核心卖点，制作阶段由分卷逐本录入天然约束剧透，不设严格剧透闸门（v0.2 §5）。
- 人工不再逐条点击候选；主操作面是异常队列 + Change 审计 + 整批回滚。
- 第一阶段继续优先使用 JSON/JSONL 和 Markdown。

## 待讨论问题

- 人工复核的真实最小操作单位是否应从“candidate”上移到“block 批处理决策”或“scene / range 批处理建议”。
- AI 输出是否应先聚合成 block-level digest，再由人工一次性确认一组低风险结构化写入。
- 哪些数据类型可以批量确认，哪些必须单条确认。
- 如何设计“自动草稿 -> 人工批注 -> Agent 结构化落盘”的低摩擦流程。
- 工作台是否应以任务流、差异视图、批处理确认和异常队列为核心，而不是候选卡片列表。
- 如何展示上下文预算和作业范围，使其服务决策而不是变成调试噪声。
