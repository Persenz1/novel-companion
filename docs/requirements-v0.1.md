# 长篇系列小说交互式阅读增强系统需求定义 v0.1

> 说明：本文档为历史需求草案，保留讨论脉络。
> 当前实现以 `requirements-v0.3.md` 及各分规格文档为准。
> 如本文档与后续规格冲突，以后续规格为准。

## 1. 项目定位

本项目定位为面向长篇系列小说的本地交互式阅读增强系统。

它不是普通 EPUB 阅读器、在线书城、百科站或单一作品专用工具。基础阅读器只是入口，核心价值是围绕长篇叙事文本提供防剧透上下文、人物卡、说话人标注、剧情线记忆、事件回顾和 AI 辅助数据整理。

目标作品包括但不限于轻小说、长篇网文、视觉小说文本、多卷系列小说、人物关系密集作品和设定密集作品。

当前需求讨论主要以《欢迎来到实力至上主义的教室》作为参照样本，因此会频繁提到班级、OAA、点数、特别考试、人物关系等具体形态。但这些内容只是用于压测和启发系统能力，不代表系统要做成单作品专用工具。最终产品形态、实体体系、数据模型和阅读增强能力都应面向通用增强阅读器设计，允许不同作品按需启用或扩展自己的实体类型、字段模板和展示模块。

## 2. 产品形态

系统定位为本地桌面 App。

长期不考虑：

- 纯 Web 服务
- 账号系统
- 云同步
- 在线内容平台
- RSS / 索引 / 订阅源
- 自动下载数据包
- 移动端优先
- 浏览器插件优先

系统支持本地导入书籍、本地导入或生成增强数据包、本地阅读、本地数据工作台整理，以及由用户自行配置的 AI 接口。

## 3. 发行与内容边界

软件发行方只发行：

- 软件本体
- 数据格式规范
- 本地解析与导入工具
- AI 提示词和 agent 工作流
- 校验器
- 数据包编译工具
- 公版、授权或自制样例数据

软件发行方不发行：

- 版权小说原文
- 可还原版权原文的大规模 parsed blocks
- 官方版权作品完整增强数据包
- 集中托管的系列百科数据
- 数据包索引站、订阅源或在线市场

具体系列、具体书籍的数据可以由发行方个人制作，也可以由用户或第三方制作。软件只支持手动导入本地文件和本地数据包。

第三方数据包导入时可以显示简短声明：数据包由用户或第三方提供，不代表软件官方内容；请确认拥有对应文本或使用权；数据包可能包含剧透、错误或不完整信息；软件只做格式校验，不保证内容合法性、准确性或完整性。

## 4. 实验样本

早期完整验证样本选择《欢迎来到实力至上主义的教室》已出版内容。

该样本仅用于验证复杂长篇系列的功能覆盖度，包括人物关系、班级与阵营变化、对话说话人、隐藏信息、防剧透角色卡、跨卷回顾、OAA、点数、特别考试等能力。系统设计不绑定该作品。

基于该样本沉淀出的字段和交互，如果明显属于《实教》特有能力，应以系列配置、实体子类型、卡片模板或可选模块的形式存在；如果具有普遍价值，则上升为通用增强阅读器能力。系统目标始终是服务多种长篇叙事作品，而不是固定服务某一本书或某一个系列。

## 5. 基础阅读器与增强层

系统包含两个相对解耦的部分：

1. 基础阅读器
2. 交互式阅读增强层

没有匹配增强数据包时，系统作为普通本地阅读器使用。有匹配且通过格式校验的数据包时，交互式阅读增强能力启用。

基础阅读器长期支持：

- TXT / EPUB / Markdown
- 本地书库
- 系列、卷、章节管理
- 目录跳转
- 搜索
- 阅读进度
- 字体、字号、行距、主题
- 滚动阅读
- 数据包匹配状态
- 增强面板显示或隐藏

暂不考虑：

- 书签
- 高亮
- 笔记
- 批注
- 划线
- 个人摘录管理
- 阅读统计
- 社交分享

## 6. 阅读模式

系统长期包含三种阅读模式：

1. 初读模式：默认模式，严格只显示当前阅读位置之前的信息。
2. 回顾模式：用户指定已读范围，例如读第 8 卷前回顾第 1-7 卷。
3. 二刷模式：明确开启后允许显示全系列信息，包括伏笔回收、真相解释、未来关系变化。

数据制作、AI 候选复核、冲突处理不作为阅读模式，而作为独立的数据工作台能力存在。

二刷模式使用独立可见边界，不受初读 read_boundary 限制。默认可以显示全系列已整理信息，也可以让用户选择只看到某卷末或某个指定边界。二刷模式必须由用户明确开启，并给出清晰提示；关闭后恢复初读模式的 read_boundary 和防剧透过滤。

## 7. 数据包类型

系统长期支持两类包：

### 7.1 注释型增强数据包

- 不包含完整正文。
- 包含结构化映射、实体、事实、关系、事件、说话人、剧情线、摘要、可见边界等增强数据。
- 需要匹配用户本地已有书籍文件。
- 更适合版权作品的公开数据交换边界。

### 7.2 完整增强书包

- 可以包含结构化正文、资源、排版信息和增强数据。
- 适合用户自用、公版文本、授权文本、自制文本，或不公开分发的个人实验。
- 实操上更方便，因为 block、正文、增强数据天然一致，不容易出现文件版本不匹配。

无论哪种包，内部增强数据 schema 应保持一致。

从用途上区分工程包和阅读包。工程包用于数据制作、AI 清洗、人工复核和后续维护，可以包含 Source、Parsed、Candidates、Review、OpenQuestions、Reports、Accepted 和 Compiled。阅读包用于普通阅读，只应包含必要的 Parsed/正文资源、Accepted、Compiled、manifest 和必要资产，不默认包含 Candidates、Review、OpenQuestions 或 AI 中间产物，避免剧透、噪音和误用。

当前阶段优先支持包含正文和必要资源的完整阅读包。阅读包内的 Markdown 主文本、图片锚点、Accepted 数据和 Compiled 查询产物应天然一致，避免用户再用外部 EPUB 进行复杂匹配。注释型阅读包和“匹配用户本地 EPUB”的能力长期保留为可能方向，但不是早期目标；尤其不优先处理不同来源、不同版本、不同排版 EPUB 的自动匹配问题。

完整阅读包是否可以公开分发取决于文本和资源权利状态。manifest 必须明确 contains_text、contains_assets、rights_note、usage_scope 等字段。软件本身支持完整阅读包格式，但公开发布时仍应区分公版、授权、自制、自用和第三方来源内容。

插图、封面、地图、人物图等资源不应只依赖运行时重新解析原始 EPUB。原始 EPUB 或原始导入文件应作为 Source 保留；清洗和解析阶段应提取资源、生成稳定资源 ID，并把图片位置锚定到章节、block 或 block 间位置。完整增强书包可以保存图片资源和锚点；注释型增强数据包原则上不分发版权图片，只保存资源指纹、位置锚点、替代说明或对用户本地资源的匹配信息。

清洗后的主文本可以优先采用 Markdown。Markdown 作为私有工程格式的目标是阅读等价、易清洗、易 diff、易交给 AI 处理、易绑定 block 和资源锚点，而不是逐像素还原 EPUB 的 CSS 排版。原始 EPUB 仍作为 Source 保留；如果未来需要接近原版排版，可从 Source 或保留的 HTML/CSS 重新渲染。早期阅读器可以使用 Markdown 渲染器承载正文、标题、分隔符、强调、脚注、图片占位和 block 锚点，不必先实现完整 EPUB 渲染器。

当前阶段阅读端优先使用 Markdown 私有正文格式。EPUB 保留为原始 Source 和未来二次开发方向，不作为早期阅读增强能力的实现前提。等 Markdown 路线跑通、数据模型稳定、工作台和防剧透查询验证完成后，再考虑 EPUB 原版渲染与 block mapping 的二期支持。

## 8. 主文本与中日参考

系统以中文文本为唯一主轴。

- block、章节、当前位置、防剧透边界、实体来源、事件来源、关系来源均以中文文本为准。
- 日文不建立独立 block 体系。
- 日文只是绑定到中文 block 上的参考内容。
- 阅读时提供开关：只显示中文，或中文 + 日文上下横版对照。
- 如果某个中文 block 没有可靠日文对应，则不显示日文参考。
- 人名、术语、组织名可以保留日文名字段，但知识系统仍以中文主数据为准。

插图资源同样绑定到中文主轴。图片不建立独立阅读进度体系，而是通过图片锚点绑定到卷、章节、block 或两个 block 之间的位置。单模态 AI 处理正文时，可以只看到图片占位符、人工标题、OCR 文本或说明文字；图片本体保留在资源区，供阅读器展示、未来多模态模型处理或人工复核使用。

中日对照允许一对一、一对多和多对一绑定。清洗阶段可以采用对齐组标记方式，让中文段落组和日文段落组共享同一个 alignment_id；一个对齐组内可以包含一个或多个中文 block，以及一个或多个日文参考段落。AI 清洗输出应遵循固定模板，明确 alignment_id、中文 block、日文参考、对齐置信度和待复核状态。阅读时以中文 block 为主轴，只有存在可靠对齐组时才显示日文参考。

## 9. 防剧透边界

防剧透边界采用混合粒度，但以 block 为核心定位单位。

- 阅读时，防剧透边界以当前 block 为准。
- 系统只展示当前 block 之前已经可见的信息。
- 当前 block 内的信息是否立即可见按类型判断，默认保守。
- 回顾时，可以按卷、章节或用户指定范围定义可见边界。
- 二刷模式下，可以允许全系列信息。
- 所有事实、关系、事件、角色状态、术语解释、剧情线摘要都必须带有 visible_from 或等价字段。
- visible_from 最好指向 block；如果数据只精确到章节或卷，也必须明确标记精度。

防剧透不是 UI 功能，而是数据查询层的硬约束。UI 只负责展示查询结果。

阅读器通过正文视口中的内部标尺判断当前 block。默认可采用屏幕中线附近的 block 作为当前阅读位置；该标尺属于内部判定机制，不需要在普通阅读界面显式显示。

系统区分当前阅读位置 current_block 和已确认阅读边界 read_boundary。current_block 表示用户现在看到哪里，用于当前场景、当前人物和当前位置相关展示；read_boundary 表示系统确认读者已经安全读完哪里，用于防剧透查询。visible_from 采用“读完该位置后可见”的语义，查询时以 read_boundary 为准，而不是直接使用 current_block。

正常连续滚动阅读时，read_boundary 可以随阅读标尺推进；目录跳转、搜索跳转、大幅拖动或跳到超过已读边界的位置时，只更新 current_block，不自动扩展 read_boundary。此时增强面板保持保守，只显示 read_boundary 之前可见的信息；用户可以手动确认“标记此前为已读”或切换回顾/二刷模式来改变可见边界。当前 block 内的信息默认不自动展示；如需在同一 block 内展示，必须由数据显式标注。

read_boundary 的自动推进只在普通连续滚动阅读中触发。第一阶段可以采用中庸的停留时间阈值：当某个 block 正常越过内部阅读标尺，并在合理时间内没有被高速滚动跳过时，才认为该 block 已读完。具体停留时长属于交互手感参数，不在需求中固定，开发时可先设置保守默认值，再通过实际阅读测试逐步缩短或调整。目录跳转、搜索定位、大幅拖动和快速滚动不应自动推进 read_boundary。

当用户跳转到超过 read_boundary 的未读后文时，右侧增强面板仍以 read_boundary 作为防剧透过滤边界。界面可以显示当前章节、当前位置和基础导航信息，但不展示该位置之后才可见的增强内容。此时应给出克制提示，让用户意识到当前位置超过已读边界，并提供“标记至此为已读”或切换回顾/二刷模式的入口。

所有涉及剧透、阅读进度、状态变化和有效范围的数据应共享同一套叙事时间线系统。该时间线以中文主文本的 series / volume / chapter / block 顺序为基础，必要时允许 scene、volume_end、chapter_end、external、manual 等辅助节点。visible_from、valid_from、valid_until、source_span、关系变化位置、事件发生位置、角色卡卷末版本、回顾触发位置都应引用这套统一时间线，而不是各自定义不可比较的位置格式。

时间线位置 ID 应可读、稳定、可排序。JSON/JSONL 中可以使用字符串表示，例如 `v01.c03.b0042`，带系列命名空间时可使用 `series_id:v01.c03.b0042`。特殊节点可使用 `v01.start`、`v01.end`、`v01.c03.start`、`v01.c03.end`、`external:guidebook01:p012`、`manual:note_001` 等形式。底层编译或数据库中可以拆分为 series、volume、chapter、block、kind 等字段以便比较和查询。

## 10. 实体类型

系统内置基础实体类型，但允许按作品或系列动态扩充。

基础类型包括：

- 人物
- 组织
- 地点
- 术语
- 能力
- 道具
- 事件
- 阵营
- 世界观设定

每个系列可以只启用自己需要的类型。未启用类型不出现在该系列的数据整理、AI 提示词、复核界面和阅读展示中。

AI 在整理文本时，如果发现现有类型无法覆盖的新概念，只能提出新增类型候选。新类型候选不能自动进入正式 schema，需要人工确认。

针对《实教》，道具、特殊能力、装备等类型可以不启用；人物、班级/组织、阵营/小组、考试/规则、积分/点数、OAA 等更重要。

实体合并必须保守。AI 可以提出“疑似同一实体”候选，但不能自动合并正式实体；自动归一只适用于来源明确、无歧义的别名或简称。人物重名、称呼模糊、代称归属不清、身份伪装或隐藏身份相关内容必须进入 Review。合并后保留 aliases、来源、变更记录和 merged_into 标记；被合并实体不物理删除，避免破坏历史引用和审计链路。

## 11. 角色卡与实体卡

角色卡采用混合式：默认是当前阅读辅助，展开后是防剧透百科。

角色卡内容分为展示摘要和结构化依据两层。展示摘要可以由 AI 起草，也可以由人工编辑，用于提供一句话印象、当前读者应该记得的背景、阅读提示和简短人物说明；但生成时必须受可见边界控制，只能基于截至目标阅读位置可见的 Accepted 数据、角色相关事实、事件、关系、剧情线和来源摘要。结构化字段如班级、身份、状态、关系、参与事件、OAA、点数等必须能追溯到 Fact、Event、Relation、SourceSpan 或对应变化记录。

AI 生成角色卡摘要时，不应直接读取全系列自由发挥，而应通过结构化检索获得人物上下文，包括该人物当前可见状态、别名、所属组织、重要事件、关系变化、相关剧情线、未解决问题和风险提示。生成结果进入 Candidate 或 Review，经人工确认后才能成为正式角色卡展示内容。角色卡展示摘要原则上按卷更新，不按每个 block 生成独立版本，避免数据量失控；人物变化、剧透事实、关系变化和事件仍然可以按 block 记录可见边界，并由阅读器查询时过滤。

卷内阅读时，角色卡默认显示上一稳定节点的摘要，通常是上一卷末或已确认的当前卷阶段摘要。当前卷内新增的事件、关系变化、身份变化和重要提示不立即重写整张角色卡，而是在“本卷新增”“最近变化”或当前相关事件模块中展示。当前卷整理完成并复核后，再生成新的卷末角色卡摘要。

人物卡至少支持：

- 头像/画像
- 姓名
- 别名、称呼、外号
- 所属班级
- 所属小组/阵营
- 截至当前位置的身份
- 当前状态
- 最近登场位置
- 初登场位置
- 当前读者应该记得的背景
- 此前做过的重要事情
- 参与过的关键事件
- 与当前场景人物的关系
- 与主角/核心人物的关系变化
- 隐藏信息状态：未知、疑似、已揭示、二刷可见
- OAA/能力评价等结构化数据
- 相关剧情线
- 相关规则/考试/事件
- 数据来源和可信度

针对《实教》，还应支持班级归属变化、学年/班级、OAA 各项评分、综合评价、班级点数/个人点数相关记录、特别考试表现、人际关系网、重要对话/发言记录、重点人物/边缘人物/临时登场人物标记。

## 12. 说话人补全

阅读界面只提供简洁标注：

- 名字模式
- 头像模式
- 头像 + 名字模式

不做：

- 多候选人展示
- 置信度数字展示
- “A？/B？”式阅读标注
- 不确定时强行标注

拿不准就不显示，进入复核，由人工判断后补全。

数据层可以保留低置信候选和待复核状态，但这是数据工作台的内容，不打扰普通阅读。

说话人标注不只支持人物实体，也应支持旁白、未知、群体发言、系统文本、歧义候选等基础类型。普通阅读界面只显示人工确认或高置信的简洁标注；未知、歧义和多候选信息只进入数据工作台复核。随着作品推进，如果出现基础类型无法覆盖的新说话来源，可以由 AI 或人工提出新增 speaker 类型候选，但必须经人工确认后进入系列配置，不能自动修改正式 schema。

## 13. 事件与剧情线

系统支持事件记录和跨卷剧情线追踪。

事件需要记录：

- 发生在哪个系列、卷、章节、block 范围
- 涉及的人物、组织、地点、规则、术语
- 事件摘要
- 对人物状态、关系、剧情线的影响
- 重要性等级
- 来源和可见边界
- 是否人工确认

事件重要性第一阶段采用四档：critical、major、minor、background。critical 表示主线推进、身份揭示、重大考试结果、核心关系变化等需要优先展示和进入卷总结的事件；major 表示对人物、关系、数值或剧情线有明确影响的重要事件；minor 表示局部有记录价值但普通阅读不一定展示的事件；background 表示背景信息、日常补充或低优先级依据。重要性等级用于展示、回顾和整理优先级，不作为文学价值评价。AI 可以建议等级，但人工可以修改。

剧情线可以包括：

- 班级竞争线
- 特别考试线
- 主角隐藏实力线
- 学生会线
- 人物个人成长线
- 阵营变化线
- 未解决谜团线

普通阅读时只展示当前相关的事件和剧情线；详情页可以查看人物时间线、剧情线时间线、伏笔列表和事件来源。

剧情线记录应尽量基于可验证事实，避免把“成长”“动机”“伏笔意义”等高度主观解释提前写成正式结论。人物成长线、伏笔线、谜团线可以作为追踪容器存在，但其节点应绑定到明确事件、关系变化、状态变化、文本暗示或后续揭示。伏笔和谜团通常以后记前：只有读到后文确认某处是伏笔或某个谜团被揭示后，才能回填前文对应位置，并严格设置 visible_from，避免初读时提前暴露解释。

## 14. 关系系统

系统支持关系图谱和关系详情模块。

关系图谱采用类似 Obsidian 的节点-连线图，展示人物、组织、班级、阵营、事件之间的关系。

关系图是实体关系图，不只限于人际关系图。第一版阅读面板中的关系图应控制为当前上下文局部图，默认包含当前场景人物、这些人物的一跳关系、当前相关事件、相关组织/班级/小组/阵营，以及必要时的考试/规则节点。复杂多方关系优先通过 Event 节点表达，不强行压成两两人物关系。

全系列大图、全量人物关系网和复杂自动布局不作为第一阶段目标。人物详情页可以提供两人关系视图，用于查看当前客观关系、历史变化、相关事件，以及受防剧透控制的主观视角、误解、隐藏动机等信息。

多方关系不强行压成两两关系，可以通过事件节点、组织节点、小组节点、剧情线节点表达。

主关系图只展示相对客观、读者当前位置已知的信息，例如同班、合作、敌对、竞争、怀疑、保护、利用、指挥、从属、交易、共同参与某事件。

每条关系需要支持：

- 起始位置
- 变化位置
- 变化原因
- 变化前状态
- 变化后状态
- 来源 block
- 影响事件
- 可见边界

多视角关系支持，但不放在主关系图上。主观视角、误解、隐藏动机、表里关系放入两人关系详情模块。

## 15. 术语、规则、OAA 与点数

默认采用术语卡，不把所有设定都做成复杂结构化系统。

普通术语卡用于学校制度、班级规则、常见设定、作品内专有名词、组织说明、地点说明、普通概念解释。

特别考试可以单独做详细卡，包括：

- 考试名称
- 所属卷/章节
- 参与班级/人物
- 规则说明
- 胜负条件
- 奖励
- 惩罚
- 过程摘要
- 结果
- 对班级/人物/关系/剧情线的影响
- 来源和可见边界

OAA/能力评价做成随阅读位置变化的实时结构化数据。只有文中明确提到，或说明书、附录、官方资料明确给出时才更新。不凭 AI 推测评分。

个人点数、班级点数等做成实时可视化数据，记录当前可见数值、变化历史、变化原因、关联事件/考试、来源位置和可见边界。

数值类客观数据应单独建模，不只作为普通 Fact 附带文本保存。第一阶段可以设置 Metric 和 MetricChange 数据区，用于记录 OAA、班级点数、个人点数、考试分数、排名、金额、人数等可量化信息。不是所有角色或组织都必须拥有数值数据，但凡文本、附录或官方资料明确给出的数值，应尽量记录。

数值数据规则更严格：不允许 AI 推测；每次数值记录或变化都必须有来源、可见边界、发生位置、关联事件或原因说明。能确定 old_value、new_value、delta 时应结构化记录；如果只知道发生变化但无法确定具体值，则只记录变化事实或待复核项，不生成精确当前值。当前数值由时间线上的 MetricChange 编译计算，并在阅读界面显示截至位置和来源。

数值数据应支持基础可视化，例如个人点数变化折线图、班级点数折线图、OAA/能力评分雷达图或分项趋势图。图表只展示当前阅读边界内可见、来源明确的数据；缺失值不插值、不推测。

## 16. 回顾功能

系统支持：

- 卷前回顾：每卷有一份总结，打开新卷或选择回顾模式时查看。
- 人物回顾：某个人物隔较长时间重新出现时，提示此前相关信息。时间跨度由人工设置或确认。
- 剧情线回顾：某条剧情线重新被提及时，回顾之前进展。
- 考试/事件回顾：再次提到某次考试、约定、冲突、结果时，回顾当时发生了什么及其影响。

不做按用户现实时间组织的继续阅读回顾。

所有回顾都受防剧透范围限制。

人物重新登场回顾以人工标记为主，系统和 AI 可以提出候选，但不应仅凭机械间隔自动打扰阅读。默认可参考卷跨度、人物重要度、上次有意义出现位置和当前场景相关性，但最终是否需要回顾由人工确认或系列配置决定。

## 17. AI 数据生产

AI 可以承担以下候选生成工作：

- 实体识别
- 别名归一
- 说话人推断
- 事实抽取
- 事件抽取
- 关系变化抽取
- 剧情线归纳
- 卷/章节总结
- 角色卡摘要生成
- 术语解释生成
- 中文 block 对应日文参考内容的对齐候选
- 新实体类型候选
- 剧透、隐藏身份、伏笔风险提示

AI 输出全部进入 Candidates 候选区。AI 不能直接修改正式数据，不能直接污染 reader 展示数据，不能直接修改原文。

AI 输出必须带来源、证据、置信度、可见边界、任务类型、模型信息。拿不准时允许输出不确定，不能硬编。

AI 工作流不要求全部发生在应用内嵌 AI 中。数据清洗、格式修复、中日对照预处理、章节结构整理、block 切分校正等前置工作，可以由用户在外部使用更强的通用 AI 工具、脚本或人工流程完成。应用内 AI 更适合在已清洗的纯文本、Markdown 或结构化中间结果上执行可复核的候选生成任务。

为了控制上下文和提高质量，AI 整理可以拆成多个独立窗口或任务阶段：

1. 数据清洗与正文规范化：得到干净的中文主文本、章节结构和必要 Markdown 标记。
2. 中日对照与 block 对齐：在独立任务中生成日文参考绑定候选。
3. 整卷粗读与结构概览：生成卷摘要、章节摘要、主要事件、主要人物、风险点和疑似剧情线。
4. 章节或 Scene 精整理：生成实体、说话人、事实、事件、关系变化、术语、数值变化等候选。
5. 跨卷连续性检查：把本卷候选与整理上下文对照，发现矛盾、遗漏、关系断裂、未解决问题推进和剧透风险。
6. 候选复核与入库：按候选类型展示，并允许查看相关段落、来源证据、历史上下文、接受、修改、拒绝或合并。

各阶段之间通过文件、结构化中间结果和 Candidates/Reports 传递，而不是依赖单个 AI 会话保留全部上下文。

AI 候选输出阶段不以 Markdown 报告作为人工阅读入口，而应直接输出规范 JSONL 或可无损转换为 JSONL 的结构化结果。人工不直接阅读候选 JSON，而是在数据工作台中通过图形化界面查看候选、相关 block、来源证据、历史上下文和可执行操作。Candidates JSONL 应按 source_span 或 block 时间线顺序存储，即使存在倒叙、回忆或插叙，也以正文 block 顺序作为复核顺序。所有正式导入 Candidates 的数据都必须通过同一套硬校验规则。

AI 数据整理默认按单卷或用户指定的局部范围执行。为了控制上下文长度，AI 不假设能够看到全系列全文；一次任务通常只读取当前卷、当前章节、当前 block/scene 范围，以及由系统提供的过往整理结果。过往整理结果应作为可检索、可压缩的工作记忆提供给 AI，包括已确认实体、别名、角色简档、关系当前状态、重要事件摘要、剧情线进度、术语解释、数值状态和未解决问题。AI 使用这些历史整理结果保持信息连续，但不能把未提供的原文内容当作已知事实。

过往整理结果需要区分两种用途：一是供阅读器展示的防剧透正式数据，二是供 AI 整理下一卷时使用的制作上下文。制作上下文可以比阅读界面更结构化、更压缩，但仍必须来自 Accepted 数据或明确标注的待复核候选，不能混入未经标记的模型臆测。

制作上下文不是无限增长的全系列摘要，而应作为可编译、可检索、可裁剪的整理上下文。对于类似《实教》这种已出版全文约数百万字、单卷约十万字量级的系列，目标是将截至当前卷前的历史整理背景压缩到约 100k-150k 的上下文量级，再与当前卷正文、当前任务范围和必要证据片段组合使用。即使使用长上下文模型，系统仍应保留结构化索引、语义检索补充和 SourceSpan 证据回查能力，避免把前序所有原文或所有整理记录无差别塞入模型。

## 18. 人工复核

系统假设增强数据制作时，人会按卷跟读一遍，因此关键数据都可以进入人工确认流程。

人工重点复核：

- 低置信说话人
- 新实体是否成立
- 别名、外号、代称归属
- 人物关系变化
- 隐藏身份、真实动机、表里关系
- 重要事件摘要
- 卷总结和剧情线总结
- 可能剧透的事实
- OAA、点数等数值数据
- AI 发现的新实体类型
- 日文参考对齐候选

系统不追求无人自动生成完整百科，而是用 AI 降低整理成本，用人工跟读保证最终质量。

## 19. 数据工作台

数据工作台是给制作者跟读整理用的可视化 Agent 界面。它不是普通阅读模式，也不是纯聊天框。

数据工作台沿用阅读界面的三栏心智：

- 左侧：系列/卷/章节目录、当前整理进度、复核队列、AI 任务列表、本章候选数量、未处理项入口。
- 中间：当前章节正文、block、scene、POV、说话人标注候选、当前 block/scene 高亮、日文参考对齐检查开关。
- 右侧：AI 候选实体、说话人、事实、事件、关系变化、剧情线、术语解释、OAA/点数变化、日文参考对齐、角色卡/术语卡编辑、接受/修改/拒绝/合并操作、来源绑定、备注与人工确认。

核心原则：数据工作台不是脱离正文的数据表，而是跟随阅读位置，把右侧增强面板换成 AI 候选和人工校验面板。

人工复核主流程按 block 顺序推进，保证制作者的阅读记忆和正文上下文连续。每个 block 或 scene 展示与当前位置相关的候选、来源、历史上下文和可执行操作；处理完成后再前进到后续 block。按候选类型筛选、批量查看说话人/事件/关系/术语等队列可以作为辅助视图，但不能替代按正文顺序跟读复核的主流程。高风险候选必须回到对应 block 上下文中确认。

数据制作不假设一次完成。每个 block、scene、章节和卷都应支持复核状态和进度断点，例如未处理、AI 已生成候选、人工复核中、已复核、有未决问题、跳过/暂不处理。系统应能从上次断点继续，统计当前卷复核进度，并清楚区分已可信区域和仍需处理区域。

人工当下无法判断的问题不应强行写入正式事实，而应进入未决问题或复核项。未决问题用于记录说话人不确定、动机不明、疑似伏笔、关系变化证据不足、别名归属不明、事件影响待确认、日文对齐不确定等内容。每条未决问题应保留来源位置、当前判断、风险等级、关联实体/事件/剧情线、需要后续回查的位置或条件。后续读到新信息时，可以回到未决问题进行确认、关闭、升级为正式数据或标记为误判。

## 20. 核心数据对象

当前核心对象包括：

- Series
- Volume
- Chapter
- Block
- Scene
- POV
- Entity
- Fact
- Relation
- Event
- Arc
- SpeakerLabel
- TermCard
- CharacterCard
- ReviewItem
- Candidate
- AcceptedData
- DataPack
- SourceSpan
- Asset
- AssetAnchor

不设置独立的提及对象。不要求记录每个人物、术语或地点在每个 block 中的所有出现。系统只记录有知识价值或交互价值的内容，例如首次登场、说话人标注、事件参与、关系变化、事实更新、术语解释入口和需要复核的出现。普通全文搜索、点击索引和临时高亮可作为编译产物或解析索引处理，不进入核心业务模型。

Fact 不表示永恒真理，而表示在某段叙事时间内成立的已确认陈述。与剧透和进度相关的 Fact 必须支持 valid_from、valid_until、visible_from、source_span、status 和变更记录。人物所属、身份状态、数值、组织成员关系、已知/未知状态等都不应通过覆盖旧值表达，而应通过新事实或变化记录表达；当前可见事实由统一时间线和查询边界计算得出。

第一阶段核心字段采用最小可用模板，不追求一次性覆盖所有未来需求。优先跑通以下对象：

- Block：id、volume_id、chapter_id、order、kind、text、review_status。
- Entity：id、type、name、aliases、first_seen、tags、status。
- Fact：id、subject_id、predicate、value、valid_from、valid_until、visible_from、source_span、status。
- Event：id、type、title、summary、position、participants、related_entities、importance、visible_from、source_span。
- RelationChange：id、relation_id、entities、relation_type、before、after、event_id、valid_from、visible_from、source_span。
- Candidate：id、type、source_span、visible_from、confidence、status、model、task_id、payload。
- CharacterCard：id、entity_id、volume_id、version_position、short_summary、reader_memory、source_refs、visible_from。
- Metric：id、subject_id、metric_type、unit、value_type、status。
- MetricChange：id、metric_id、old_value、new_value、delta、reason_event_id、valid_from、visible_from、source_span。
- Change：id、operation、candidate_id、target_type、target_id、before、after、reason、source_span。

所有正式对象应尽量包含 schema_version、series_id、status、source_span、visible_from、created_by、updated_at 等通用字段；但第一阶段允许按对象性质裁剪。字段设计以可追溯、防剧透、可复核、可编译为核心，不为了百科完整度提前引入过多细项。

## 21. Scene 与 POV

Scene 表示一段连续 block 构成的语义场景，例如教室对话、宿舍谈话、特别考试说明、小组行动、学生会会议、人物独白、回忆片段等。

Scene 用于帮助推断说话人、展示当前场景人物、提供右侧上下文、辅助事件抽取、辅助关系变化判断，以及处理《实教》这种频繁切换地点、人物组合和对话场景的作品。

Block 是硬定位单位，必须稳定存在；Scene 是语义增强单位，不作为导入和基础阅读的强制门槛。没有 Scene 时，系统仍应能够完成基础阅读、block 定位和基础增强查询；有 Scene 时，系统可以获得更好的说话人推断、当前场景人物展示、事件抽取和关系变化判断。一个 Scene 默认覆盖连续多个 Block；一个 Block 默认只归属一个主 Scene，避免复杂交叠。

Block 切分以自然段落为主。对话、旁白段落、独立说明段落通常各自成为一个 block；不默认切到句子级，也不把整章或整节作为单个 block。数据清洗阶段可以人工或借助外部工具合并、拆分异常 block，但一旦进入正式 Parsed 数据，block ID 和顺序应保持稳定。

POV 用于记录叙述视角。它可以作为 Scene 或章节段落的字段，不一定需要做成重型独立对象。POV 不要求在每个 Scene 中重复记录，只在叙述视角发生切换、进入回忆/插叙、出现特殊叙述结构或存在隐藏身份、误导叙述风险时显式记录。POV 用于区分当前内容来自谁的视角、是否是第一人称独白、是否是第三人称旁观、是否是回忆/插叙，以及辅助隐藏身份、误导叙述、内心独白处理。

## 22. 来源追溯

所有重要增强数据都必须能追溯来源，包括事实、关系、事件、剧情线节点、角色卡内容、术语卡内容、特别考试规则、OAA 评分、点数变化、说话人标注、卷总结、章节总结、人物回顾和剧情线回顾。

来源追溯和防剧透可见边界必须分开建模。SourceSpan 回答“这条信息从哪里来”，visible_from 回答“读者最早读到哪里可以安全看到这条信息”。二者经常相同，但不能合并；例如伏笔暗示、延迟揭示、整卷总结、外部资料和人工录入都可能拥有不同的来源范围和可见边界。

来源可以指向：

- 系列
- 卷
- 章节
- Scene
- Block
- 外部资料/说明书/附录
- 人工录入说明

第一阶段 SourceSpan 精度到 block 或连续 block range 即可，不支持也不要求 block 内字符范围。block 本身以自然段落为主，粒度已经足够复核、证据回查和防剧透定位；不引入字符 offset，避免增加清洗、编辑和校验复杂度。

没有来源的重要信息不能进入正式增强数据；来源不清的内容只能作为候选或备注。

## 23. 数据区划分

系统长期区分以下数据区：

- Source：原始导入书籍，或完整增强书包中的正文与资源。
- Parsed：从 Source 得到的结构化正文，包括系列、卷、章节、block、scene、POV、资源引用等。
- Candidates：AI 或规则生成的候选数据。
- Accepted：已通过人工确认或校验规则的正式增强数据。
- Review：待人工处理的复核队列。
- Compiled：供阅读器高效查询和展示的最终数据。
- Reports：解析报告、AI 任务报告、校验报告、质量报告、缺漏报告。

AI 不能直接写 Source、Parsed、Accepted 或 Compiled。

Accepted 数据的变更必须有记录。AI 候选经人工接受后，不应以无痕覆盖的方式直接改写正式数据，而应生成类似 git diff 的变更记录，说明新增、修改、合并、弃用、冲突解决或状态变化的内容、原因、来源、操作者和时间。对于关系、状态、点数、身份揭示、剧情线进度等会随阅读位置变化的数据，应优先以追加变化记录的方式保存，当前状态由查询或 Compiled 产物根据可见边界计算得出。

删除正式数据时默认采用弃用或 tombstone 标记，避免破坏历史追溯。角色卡等人工摘要允许更新，但也应保留修订记录、来源和可见边界。变更记录既用于审计和回滚，也用于解释当前展示状态为何成立。

工程期数据包优先采用文件夹形式组织，交换和导入时可以打包为 zip。清洗后的主文本使用 Markdown；结构化数据优先使用 JSONL；manifest、配置、报告摘要使用 JSON；图片等资源使用普通文件。SQLite、全文索引、向量索引或其他高性能查询结构只作为 Compiled 编译产物，不作为人工维护的主源格式。

推荐目录结构示例：

```text
bookpack/
  manifest.json
  source/
  parsed/
    volumes/
    blocks.jsonl
    scenes.jsonl
    assets.jsonl
    asset_anchors.jsonl
  accepted/
    entities.jsonl
    facts.jsonl
    events.jsonl
    relations.jsonl
    arcs.jsonl
    character_cards.jsonl
    term_cards.jsonl
    speaker_labels.jsonl
    changes.jsonl
  candidates/
    candidates.jsonl
  review/
    review_items.jsonl
    open_questions.jsonl
  compiled/
  reports/
```

JSONL 的目标是让大量对象一行一条，便于人工检查、git diff、脚本处理、AI 生成、增量追加和后续导入数据库。正式运行时可以从 Markdown + JSON/JSONL 编译出 reader.sqlite、搜索索引、整理上下文和其他缓存。

## 24. 数据包 Manifest

每个数据包需要包含元信息，用于识别、校验和展示。

暂定字段包括：

- 数据包名称
- 系列名
- 卷列表
- 语言：中文主轴，日文参考可选
- 数据包类型：注释型 / 完整增强书包
- schema 版本
- 适配 reader 版本
- 作者/制作者
- 生成方式：人工 / AI / 混合
- 是否包含正文
- 是否包含日文参考
- 是否包含头像/图片资源
- 来源文件指纹或章节结构指纹
- 支持的阅读模式：初读 / 回顾 / 二刷
- 数据完整度说明
- 校验报告摘要
- 版权/使用说明

Manifest 是可演进的。

## 25. 校验机制

校验机制分为硬校验和软校验。

硬校验用于数据格式、引用完整性和系统安全边界，失败时应阻止导入、编译或发布。第一层硬校验包括：

- 数据包结构是否符合 schema
- 必填字段是否存在
- 字段类型是否正确
- ID 是否唯一
- 引用的系列、卷、章节、block、scene 是否存在
- 数据包版本是否兼容
- manifest 是否完整
- 数据区划分是否正确
- 数据能否被正常读取和编译

软校验用于内容质量、事实一致性、摘要完整性、疑似遗漏、剧透风险和冲突提示。软校验不替代人工判断，结果进入 Reports 或 Review，供制作者参考。

文本事实正确性主要由逐 block 的人工跟读和复核流程保证。制作者如果对整理结果不放心，可以在人工核查后的 Accepted 数据和 Compiled 数据基础上，再运行 AI 复查任务，让 AI 对照正文、来源、变更记录和整理上下文生成疑点报告，但 AI 复查结果仍只能作为候选或报告，不能自动改写正式数据。

## 26. 阅读界面

阅读界面采用三栏思路：

- 左侧：本地书库、系列列表、卷列表、章节目录、搜索、阅读模式切换、数据包状态提示。
- 中间：正文阅读、滚动阅读、说话人名字/头像/头像+名字显示、当前 block/scene 高亮、中文主文本、日文参考开关、基础阅读设置。
- 右侧：当前场景人物、当前人物卡、当前关系、当前剧情线、当前事件/考试回顾、当前术语卡、人物重新登场提醒、相关卷总结/剧情线回顾、数据来源入口、必要时的人工修正入口。

没有匹配增强数据包时，只作为基础阅读器使用。增强面板可以隐藏。

## 27. AI 接口与责任边界

如果用户要使用数据工作台进行 AI 辅助整理，需要配置自己的 AI 接口。

系统支持：

- OpenAI API
- 兼容 OpenAI 格式的接口
- 用户自行配置 API Key、Base URL、模型等参数
- 软件不内置官方密钥
- 软件不代用户中转请求
- AI 请求由用户本机直接发起
- 用户可以不开启 AI，只使用阅读器或手工整理能力
- AI 只处理用户主动提交的章节、block 或任务范围

用户自行决定是否把本地文本发送给 AI 服务，并自行负责所使用 API 服务的条款、隐私和版权风险。

## 28. 质量原则

暂定质量原则：

- 防剧透优先于信息完整。
- 不确定时宁可不显示，也不乱标。
- AI 输出必须可追溯。
- 人工确认结果优先。
- 阅读界面不能被候选、置信度、错误提示打扰。
- 数据工作台可以复杂，阅读界面必须克制。
- 每卷整理结果应该能独立检查。
- 数据包应该能重复生成、重复编译。
- 角色、关系、事件变化要有来源。
- 数值数据不能靠推测。

## 29. 后续待讨论

本版本不设定开发优先级。后续继续讨论：

- 完整需求文档结构
- 数据对象字段
- 数据对象关系图
- AI 任务拆分
- 数据工作台具体交互
- 《实教》专属字段模板
- 数据包格式选择
- 完整增强书包组织方式
- 工程架构与实现路线

当前阶段的第一版实现重点不是发布打包优化，而是跑通完整制作闭环。第一阶段应优先规定清洗后数据接受形式，包括 Markdown 主文本格式、block 标记、图片锚点、中日对照对齐组、JSON/JSONL 中间结果和 AI 输出模板；实现数据工作台的最小可用复核流程；实现最低限度可用的 Markdown 阅读器，用于验证 block 阅读标尺、防剧透查询和右侧增强面板。等清洗、AI 候选、人工复核、Accepted 数据、Compiled 查询和阅读展示全流程跑通后，再考虑阅读包打包、导入优化、EPUB 二期支持和更完整的桌面应用体验。

第一阶段样例数据应优先使用自造测试文段，而不是直接使用正式版权文本。测试文段应刻意覆盖系统关键能力，包括章节、block、scene、人物、组织、对话说话人、事件、关系变化、数值变化、角色卡、术语解释、图片锚点、中日对照、未决问题、候选复核、变更记录和防剧透边界。目标是用小样例完整跑通清洗、解析、AI 候选、人工复核、Accepted 入库、Compiled 查询和 Markdown 阅读展示，再进入正式文本制作。

第一阶段开发应按制作流程分段测试，而不是跳到完整应用。优先从数据清洗开始，先规定并验证清洗后数据接口，再进入后续解析、候选、复核和阅读展示。推荐顺序为：清洗输入/输出模板；Markdown block/asset/alignment 标记；Parsed JSONL 生成；硬校验；AI Candidates JSONL 模板；按 block 复核工作台；Accepted/Change/OpenQuestion 入库；Compiled 查询；最低限度 Markdown 阅读器。每一段都用自造样例验证后再进入下一段，避免后期才发现前置接口缺失。

清洗阶段强制规定输出接口，不强制规定输入来源或清洗方法。上游可以是 EPUB、TXT、OCR、复制文本、外部大模型、Codex/GPT、脚本或人工整理；系统不关心清洗如何完成，只要求交付物符合 Markdown 主文本、HTML 注释标记、资源锚点、中日对齐组和必要 JSON/JSONL 接口规范。这允许用户在清洗阶段使用性能更强、上下文更大的外部模型。

blocks.jsonl、assets.jsonl、asset_anchors.jsonl、alignments.jsonl 等 Parsed 索引文件第一阶段可以视为可生成文件。清洗者只需保证 Markdown 主文本、资源文件和必要标记符合规范；解析器可以从 Markdown 和资源目录生成 JSONL 索引。如果外部 AI 或脚本已经生成 JSONL，也允许导入，但必须经过同一套硬校验。

清洗端应形成“生成 + 验证”闭环。Codex、脚本或校验器应检查清洗输出是否完全合格，包括 Markdown block 标记是否完整、block ID 是否唯一且有序、章节结构是否可解析、图片锚点是否有效、中日对齐组是否引用有效 block、生成的 JSONL 是否能通过硬校验。只有清洗端验证通过后，才进入 AI 候选生成和人工复核阶段。

第一阶段工具链应规定 AI 候选 JSONL 输出模板，优先服务实体、说话人、事实、事件、关系变化、术语、数值变化、中日对齐和未决问题等候选类型。清洗阶段可以使用 Markdown 和人工可读中间文本，也可以由 GPT/Codex 等外部工具直接输出规范 JSONL；如果外部工具输出不稳定，再使用小型转换器或解析器作为兜底。候选复核阶段则以结构化 Candidates 数据和图形化工作台为主。

清洗后的 Markdown 标记优先采用通用 HTML 注释，不设计额外私有语法。例如 block 标记使用 `<!-- block: v01.c03.b0042 -->`，图片锚点、对齐组、scene 边界等也可采用类似注释形式。HTML 注释在普通 Markdown 渲染中默认不显示，便于 AI 阅读、人工编辑和脚本解析，也不污染正文内容。

清洗后的正文文件第一阶段采用一卷一个 Markdown 文件，例如 `parsed/volumes/v01.md`。单卷约十万字量级对当前长上下文 AI 和本地处理都可接受；清洗后的文本已经去除无关噪声，按卷处理不会造成不必要的上下文膨胀。章节、block、scene、图片锚点和中日对齐组都在卷内通过 HTML 注释标记，block ID 保持全系列唯一或在系列内可唯一解析。

block ID 应在清洗输出阶段直接写入 Markdown，而不是每次由解析器根据段落重新生成。解析器可以校验、补全缺失标记、检查顺序和报告冲突，但正式 Parsed 数据中的 block ID 应保持稳定，供 SourceSpan、visible_from、valid_from、valid_until、图片锚点、中日对齐和候选数据引用。

清洗过程中、进入 AI 候选生成和人工复核前，可以重新切分段落并重排 block ID。一旦进入评估/复核流程，或已有 Candidates、Review、Accepted 数据引用这些 block，旧 block ID 即视为冻结，不能为了连续编号而重排。后续如果必须插入新 block，应使用可排序的后缀 ID，并在变更记录中说明插入位置、原因、影响范围和是否需要重新校验。

Parsed 的 blocks.jsonl 应保存 block 正文、顺序、章节归属和必要元信息。Markdown 仍是主文本来源，blocks.jsonl 是解析索引和校验依据；保留正文副本有助于 AI 任务切片、搜索和 SourceSpan 回查。第一阶段暂不强制引入 text_hash，因为当前制作流程以人工逐步清洗和复核为主，正文进入复核后默认不再频繁变动；后续如需要支持正文迁移、版本差异检测或外部 EPUB 匹配，再加入文本 hash 或相似度校验机制。
